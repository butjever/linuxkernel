#### Hardware accelerator for eBPF
#### 简要描述
eBPF 是 Linux 内核主线支持的字节码技术，允许在内核态执行非可信用户代码，为网络可观测性提供了必要的基础设施。然而 eBPF 处理网络数据包的性能受限于 OS，且无法介入 Kernel bypass 场景下网络应用的数据通路，难以充分满足高性能网络对可观测性的需求。网络硬件尚不具有通用的可编程规范，使得基于 eBPF 的网络可观测性方案难以下推至既有硬件；而通过在硬件层面实现 eBPF 字节码执行能力，即可复用 Linux 生态下的可观测性方案。本次分享将介绍硬件 eBPF 的现状与我们基于数据流架构实现硬件 eBPF 加速器的工作。

####  如何在不同内核版本进行eBPF开发？
#### 简要描述

Linux内核版本经历了3.x向4.x到5.x演进，eBPF技术的支持也是从4.x开始更加完善起来，但是云上服务器有大量的3.10内核版本是不支持eBPF的，为了让我们现有的eBPF工具在这些存量机器得以支持，享受到eBPF带来的红利，我们移植了eBPF到低版本内核，保证一个工具可运行于低、中、高的内核版本。
 另外，当前的BPF开发方式主要有：
 1、纯libbpf应用开发：这种开发方式不仅效率低，跨内核可移植难；
 2、借助BCC：开发效率高、可移植性好，并且支持动态修改内核部分代码，非常灵活。但存在部署依赖Clang/LLVM等库；每次运行都要执行Clang/LLVM编译，严重消耗CPU、内存等资源，容易与其它服务争抢。
LCC（Libbpf Compilation Collection）以 BPF CORE 为基础实现，保留了资源占用低、可移植性强等优点，还融合了BCC动态编译的特性，适合在生产环境批量部署应用。
我们通过优化eBPF工具的开发方法，节省了资源消耗，同时原来基于BCC的python工具，只需要在LCC的帮助下，就能在全量内核版本安全运行，给广大eBPF的爱好者提供一种新的探索和实践。

提纲：
1.eBPF概述和开发流程
2.libbpf CORE开发方式
3.各内核版本eBPF实践
4.LCC框架简介

收获：
1.eBPF概述及指令架构
2.eBPF CORE开发方法和资源优化方法
3.LCC框架帮助快速构建eBPF工程的能力

虽然垫片层功能简单，但它依旧引入了额外的资源消耗，即 Go 版本的垫片层大致需要 8MB 左右 VMRss，随着容器数目增加，垫层片的内存消耗也会线性增加。
当垫片层出现问题时，我们难以做到原地升级，进而增加了软件管理成本。为了解决这个问题，本次议题提出了基于 eBPF 管理进程状态的方案，通过
移除垫片层来实现 Daemon-less Container。

方案相关连接：https://github.com/fuweid/embedshim

1. 容器进程管理的发展历程

分布式系统的性能追踪是个老大难的问题。不同于IT领域的微服务调用链有比较成熟通用的调用链追踪系统。在CT领域，现有的追踪框架基本都是侵入式的，不是在业务代码上打点就是在平台或框架上打点，这种追踪方式比较大的问题就是跟具体的业务、框架有强耦合，一旦换了个产品或者切换另一种通信框架就不适用了。所以，这里希望利用Linux内核内置的eBPF动态插桩技术，做出一个非侵入式、用户无感知、无需重编重启的分布式追踪系统。

#### 基于eBPF的网络报文跟踪工具nettrace
#### 简要描述

云场景下，网络环境更为复杂。面对网络问题，常规的定位工具存在一定的短板，难以高效地跟踪网络报文在内核协议栈中的处理过程。为了解决这一问题，腾讯云虚拟化团队基于eBPF/BCC开发了内核协议栈报文跟踪工具nettrace。该工具能够深入内核来跟踪报文在协议栈中所经过的路径以及报文的整个生命周期，通过观察报文的生命周期可以一目了然地看出网络故障的原因，达到网络故障快速定位的目的。同时，本工具提供了对NAT/IPVS的支持，可以直观地观察到报文在内核中信息变动的过程。

项目链接：https://github.com/xmmgithub/nettrace

#### 作者简介
董梦龙，腾讯云虚拟化团队

#### eBPF 动态加载框架：eBPF as a Service  
#### 简要描述
对比 Web 应用程序的发展，eBPF 与内核的关系有点类似于 JavaScript 与浏览器内核的关系。eunomia-bpf 是一套 CO-RE eBPF 编译工具链和运行时，将 eBPF 的编译和运行阶段完全分离，只需要编写内核态的 C 代码，编译完成后通过 API 请求直接分发 eBPF 字节码和一些加载信息，仅需数十 kB 的请求和数十毫秒的启动时间，无需 LLVM 依赖，即可实现在不同内核版本上动态加载运行、热更新任意 eBPF 程序。eunomia 适合批量部署、动态加载、装配 eBPF 程序，相比传统的通过 BCC 或原生 libbpf 部署，启动时间和资源消耗降低了数十倍，且开发方式简单便捷。

#### 基于eBPF的Service Mesh性能瓶颈定位与优化  
#### 简要描述
伴随着容器、微服务、DevOps等新型IT技术的发展，云原生系统开始被广泛研究和落地。作为典型的云原生系统，Service Mesh和FaaS因为其进一步降低了开发和部署的代价，提升了开发和部署的效率，成为大家关注的热点话题。但是，这两种云原生系统对网络代理的依赖性较高，每一个服务实例或者函数实例即数据面均有代理相伴。网络代理复杂的数据报文处理过程不仅增加了系统资源消耗，也导致请求服务质量的下降，难以满足高并发场景中业务的需求，成为其在工业场景中大规模落地的主要障碍。因此，如何优化数据面网络的性能甚至消除网络代理成为需要迫切解决的问题。本次报告介绍一种基于eBPF的service mesh和FaaS数据面网络优化方法，并给出相关实验结果。

#### DeepFlow 基于 eBPF 的高度自动化可观测性实践 
#### 简要描述
可观测性建设在云原生时代越来越重要，通常包含指标、分布式追踪、日志三大支柱的工作，在高速迭代的软件开发过程中，由于语言、框架的差异和发展，手动打桩插码的传统方式越来越消耗着开发团队的时间。利用 eBPF 的零侵扰特性，DeepFlow 实现了高度自动化的可观测性数据采集和分析能力。在指标方面，通过 eBPF/cBPF 采集应用程序的系统、网络、应用性能黄金指标，实现全景、全栈的性能监控；在追踪方面，通过 eBPF 能力自动关联一个事务中的分布式调用，自动绘制调用链火焰图，且能自动关联一个请求经历应用代码、服务网格 Sidecar、容器网络 iptables、云网络 OvS、NFV 网关等的全栈调用，让追踪无盲点；在日志方面，通过 eBPF/cBPF 自动采集各类应用协议的访问请求并形成访问日志，包括 HTTPS 加密请求、HTTP2/gRPC 请求等。除此之外，DeepFlow 利用 AutoTagging 和 SmartEncoding 技术将这些不同类型的数据进行自动关联，使得可观测性从此告别数据孤岛。

向阳，云杉网络研发 VP，DeepFlow 开源社区负责人
